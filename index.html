CHAPTER 01

Introduction to Shopify App Development

Beginner

You already know how to build Shopify themes. You can create sections, write Liquid, add metafields, and customize storefronts. That's great — it's a real skill. But theme development has a ceiling. Shopify App Development is the next level.

🤔 What is Shopify App Development?

Think of it this way: A theme changes how a store looks. An app changes what a store can do. Apps add new features, connect to external services, automate tasks, and process data — things that Liquid alone can never do.

Shopify App Development means building software that merchants install from the Shopify App Store (or use privately) to power their business.

🎨
Theme Dev (You Know This)
Liquid templates, sections, CSS, metafields. Changes store appearance.
⚡
App Dev (New Territory)
JavaScript, APIs, databases, backend logic. Adds new store capabilities.

💰 Why App Developers Earn More

Theme freelancers typically earn ₹3,000–₹15,000 per project. App developers earn ₹30,000–₹5,00,000+ per project. Why such a big difference?

  • Apps solve real business problems — inventory sync, automated emails, custom checkout, loyalty programs
  • Apps have recurring revenue — merchants pay monthly subscriptions, not one-time fees
  • Apps require more technical skill — supply is low, demand is high
  • App Store income — your app can earn money while you sleep
📌 Real Numbers
Top Shopify apps in the App Store earn $10,000 to $100,000+ per month from subscriptions. Even small utility apps earn $500–$5,000/month passively.

CHAPTER 02

Types of Shopify Apps

Beginner

Not all Shopify apps are the same. Before you start building, you need to understand the five main types of Shopify apps — because each type is used in a different situation.

1. Custom Apps

A custom app is built for one specific merchant. The merchant requests a unique feature (like "sync my orders to my custom ERP"). You build it. Only that merchant uses it. It doesn't appear in the App Store.

Best For Freelancers
Great starting point. You get paid per project. No need to manage subscriptions or deal with App Store approval. Start here!

2. Public Apps

A public app is published on the Shopify App Store. Any merchant can find and install it. You charge a monthly subscription. This is the SaaS (Software as a Service) model — the biggest earning opportunity.

Real Example
"Klaviyo Email Marketing" or "PageFly Landing Page" — these are public apps earning millions monthly.

3. Private / Internal Apps

A private app is for internal company use. For example, a large brand builds an app just for their team to manage operations. It's never listed publicly, never installed by outside merchants.

4. Embedded Apps

An embedded app runs inside the Shopify admin dashboard. When a merchant opens it, it appears as a page within Shopify — not as a separate website. This uses App Bridge technology. Most modern Shopify apps are embedded.

Shopify Admin
→
Your Embedded App Tab
→
App UI (iframe)

Embedded App appears inside Shopify Admin as a tab/page

5. Standalone Apps

A standalone app lives on its own website — separate from Shopify admin. Merchants access it via a separate URL. Less common now because Shopify prefers embedded apps, but still used for complex tools or dashboards.

🔧
Custom App
1 client. One-time fee. Perfect for freelancers starting out.
🌐
Public App
App Store. Many merchants. Monthly subscriptions = passive income.
🏢
Private App
Internal use only. Built for one company's own operations.
📱
Embedded App
Lives inside Shopify Admin. Best UX. Industry standard.
🖥️
Standalone App
Separate website/dashboard. For very complex tools.
⚠️ Beginner Advice
Start by building a Custom App or an Embedded App for a client. Don't rush to publish on the App Store — learn the basics first. The App Store has a review process that can take weeks.
✓ Pro Tips
  • Build embedded apps — merchants prefer them
  • Study the Shopify App Store for ideas
  • Look for gaps: what do merchants complain about?
  • Custom app = fastest way to first income
✗ Common Mistakes
  • Building standalone when embedded is expected
  • Publishing to App Store too early
  • Not understanding which type fits the use case

CHAPTER 03

Understanding APIs

Beginner–Intermediate

The word "API" might feel scary. Don't worry — by the end of this chapter, APIs will feel like a totally normal thing. You've already used APIs without knowing it (Shopify sections talk to liquid objects that come from data sources — same concept).

What is an API?

Imagine you're at a restaurant. You don't walk into the kitchen and cook your own food. You give your order to the waiter. The waiter goes to the kitchen, gets the food, and brings it back to you.

An API (Application Programming Interface) is the waiter. Your app is you (the customer). Shopify's data is the kitchen. You ask the API for data, it goes and gets it, and returns the result.

Your App
→ Request
API
→ Fetch
Shopify Data
← Response
Your App

How API communication works

REST API vs GraphQL API

📡
REST API
Older style. Each URL = one type of data. Simple to understand. Shopify is moving away from REST but it still works. Example: GET /orders.json
⚡
GraphQL API
Newer, faster. You ask exactly what data you want in one request. Shopify now prefers GraphQL. More powerful. Slightly more complex to write.

GraphQL Example (Beginner Friendly)

GraphQL 
// Ask Shopify: "Give me the first 5 products with title and price"
{
  products(first: 5) {
    edges {
      node {
        id
        title
        priceRange {
          minVariantPrice {
            amount
          }
        }
      }
    }
  }
}

Key Shopify APIs

📦 Shopify Admin API ▼
The main API. Lets your app read and write store data — products, orders, customers, inventory. Requires a merchant to install your app and grant permission. This is what you'll use 90% of the time.
🛒 Storefront API ▼
Used to build custom storefronts (Headless Commerce). Lets you fetch products, create carts, and do checkout from any frontend — not just Shopify themes. Uses a public token, not OAuth.
🔔 Webhooks ▼
A webhook is a "notification" from Shopify to your app. When something happens (new order placed, product updated), Shopify sends a POST request to your URL. You don't need to keep polling — Shopify tells you automatically. Example: new order webhook fires when customer buys something.
🏷️ Metafields API ▼
You already know metafields! The Metafields API lets your app create and read custom data attached to products, orders, customers. Same concept as theme metafields, but via API from your backend.
🔐 OAuth & Access Tokens ▼
OAuth is the system that lets merchants securely authorize your app to access their store data. Think of it like "Login with Google" — when a merchant installs your app, they click Authorize, Shopify verifies everything, and gives your app an Access Token. Your app stores this token and uses it for all future API calls to that merchant's store.

OAuth Flow — Step by Step

1️⃣ Merchant clicks "Install App" in App Store
↓
2️⃣ Shopify redirects to your app's OAuth URL
↓
3️⃣ Shopify shows merchant a permission screen
↓
4️⃣ Merchant clicks "Authorize" → Shopify sends code
↓
5️⃣ Your app exchanges code for Access Token
↓
6️⃣ Store Access Token in your database
↓
7️⃣ Use token to make API calls forever ✅
⚠️ Access Token = Your Password to the Store
Never expose access tokens in your frontend code. Never log them. Store them securely in your database or environment variables. If leaked, a bad actor can access the merchant's store data.
✓ Pro Tips
  • Use GraphQL — Shopify is deprecating REST
  • Set up webhooks to avoid polling APIs
  • Test API calls in Shopify's API Explorer
  • Save access tokens encrypted in your DB
✗ Common Mistakes
  • Putting API tokens in frontend JavaScript
  • Not requesting the right OAuth scopes
  • Ignoring webhook verification (security risk)
  • Using REST when GraphQL is better

CHAPTER 04

Shopify App Architecture

A Shopify app isn't just "one thing". It has multiple parts that work together. Understanding this architecture will help you know what to learn and why each piece matters.

The 4 Layers of a Shopify App

Frontend (React / Next.js)
What merchant sees in Shopify Admin — buttons, forms, dashboards
↕ HTTP Requests
Backend (Node.js / Next.js API Routes)
Business logic, OAuth, API calls, webhooks — the brain
↕ Queries
Database (SQLite / PostgreSQL / MongoDB)
Stores access tokens, app settings, merchant data
↕ API Calls
Shopify APIs (Admin API, Webhooks)
The source of truth — products, orders, customers, etc.

Key Technologies Explained

⚛️ React / Polaris — Frontend UI ▼
Your app's UI is built in React. Shopify provides a UI component library called Polaris — it gives you buttons, cards, tables, etc. that look exactly like Shopify's own UI. This is important because embedded apps should feel native to Shopify Admin.
🟢 Node.js — Backend Runtime ▼
Node.js lets you run JavaScript on the server (backend). You're probably used to JavaScript only in the browser. Node.js takes JavaScript and makes it run on a server — handling API requests, database queries, and OAuth flows. With Next.js, you don't even need to set up Node separately.
🔷 Next.js — Full-Stack Framework ▼
Next.js is a framework that gives you React frontend + Node.js backend in a single project. One codebase, one deployment. Perfect for Shopify apps. The official Shopify CLI scaffolds apps in Next.js by default.
🌉 App Bridge — Shopify Connector ▼
App Bridge is a JavaScript library from Shopify that connects your embedded app to the Shopify Admin shell. It handles session tokens, navigation, toast notifications, and modals that feel native to Shopify.
🗄️ Database — Storing App Data ▼
You need a database to store access tokens per merchant, app settings, and any data your app creates. Shopify CLI apps default to SQLite (for local dev) and PostgreSQL or MySQL in production. You can also use services like PlanetScale or Supabase — both have free tiers.

App Request Flow

Merchant opens app
→
App Bridge sends session token
→
Your backend verifies token
→
Calls Shopify Admin API
→
Returns data to frontend
✓ Pro Tips
  • Use Polaris for all UI — looks native
  • Keep all API calls on backend (never expose tokens)
  • Use Prisma ORM for database management
  • Understand App Bridge session tokens early
✗ Common Mistakes
  • Making API calls from React frontend directly
  • Not using Polaris (UI looks inconsistent)
  • Not understanding the session token flow
  • Forgetting to verify webhook HMAC signatures

CHAPTER 05

Next.js for Shopify Apps

Intermediate

Next.js is the official framework used in Shopify CLI's app template. If you want to build Shopify apps, you need to learn Next.js. The good news? It's built on React, which you'll learn as you go. And the App Router makes backend logic feel almost magical.

Why Next.js?

  • Frontend + Backend in one project — React UI and API routes live together
  • Server-side rendering — Pages can render on the server (faster, better SEO)
  • API Routes — Build a REST API without a separate backend server
  • Shopify CLI uses it — The official scaffold generates Next.js projects
  • Easy deployment — Works perfectly on Vercel, Railway, Render

Next.js File Structure (App Router)

File Structure 
my-shopify-app/
├── app/
│   ├── layout.jsx         ← Root layout (header, providers)
│   ├── page.jsx           ← Home page /
│   ├── api/               ← Your backend API routes
│   │   ├── auth/
│   │   │   └── route.js   ← Handles OAuth flow
│   │   └── webhooks/
│   │       └── route.js   ← Handles incoming webhooks
│   └── [...routes]        ← App pages
├── prisma/                ← Database schema
├── shopify.server.js      ← Shopify auth setup
└── package.json

API Route Example — Fetch Products

Next.js API Route (app/api/products/route.js) 
// This is a BACKEND file — runs on server, never exposed to browser
import { authenticate } from "../../../shopify.server";
export async function GET(request) {
  // Step 1: Verify the request is from a real logged-in merchant
  const { admin } = await authenticate.admin(request);
  // Step 2: Query Shopify's GraphQL API
  const response = await admin.graphql(
    `#graphql
      query {
        products(first: 10) {
          edges {
            node {
              id
              title
              totalInventory
            }
          }
        }
      }
    `
  );
  // Step 3: Return data as JSON to frontend
  const data = await response.json();
  return Response.json(data);
}

Server Components vs Client Components

Key Concept: Server vs Client
In Next.js App Router, components are Server Components by default — they run on the server and can directly access databases or APIs. Add "use client" at the top when you need browser interactivity (onClick, useState, forms).
Server Component (default — no need to mark) 
// Runs on server. Can fetch data directly. No "use client" needed.
export default async function ProductsPage() {
  const products = await fetchProductsFromDB();
  return <div>{products.map(p => <p key={p.id}>{p.title}</p>)}</div>;
}
Client Component (add "use client" for interactivity) 
"use client"; // This line makes it a client component
import { useState } from "react";
export default function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>Count: {count}</button>;
}
✓ Pro Tips
  • Learn App Router not Pages Router (it's newer)
  • Use server components for data fetching
  • Keep API routes on server — never expose secrets
  • Next.js docs at nextjs.org are excellent
✗ Common Mistakes
  • Mixing Pages Router and App Router patterns
  • Making Shopify API calls in Client Components
  • Not understanding async/await
  • Ignoring loading.js and error.js files

CHAPTER 06

Shopify CLI — Complete Guide

Beginner → Intermediate

Shopify CLI is your most important tool as a Shopify app developer. It does everything automatically — creates your app, connects to your dev store, runs a local server, and handles the confusing OAuth setup for you. You don't need to set up OAuth manually anymore.

What is Shopify CLI?

Shopify CLI (Command Line Interface) is a tool you run in your terminal. It scaffolds a complete Shopify app project with all the correct setup, handles authentication for you, and creates a live tunnel so Shopify can talk to your local computer during development.

Think of it as the "create-react-app" but for Shopify apps — but much more powerful.

Installation

⚠️ Prerequisites First
Before installing Shopify CLI, make sure you have: Node.js v18+ and npm or pnpm installed. Check: node --version
Terminal — Install Node.js (if needed) 
# Download Node.js LTS from:
# https://nodejs.org
# Verify installation
node --version      # Should show v18.0.0 or higher
npm --version       # Should show 9.x or higher
Terminal — Install Shopify CLI 
# Install Shopify CLI globally
npm install -g @shopify/cli
# Verify installation
shopify version
# Should output something like: @shopify/cli/3.x.x

Login to Your Shopify Partner Account

Terminal — Login 
# Login to your Shopify Partner account
shopify auth login
# This opens your browser to log in
# Use your partners.shopify.com account

Create a New App

Terminal — Create App 
# Create a new Shopify app
shopify app init
# CLI will ask you:
# - App name
# - Template (choose: node — Next.js based)
# - Package manager (choose npm or pnpm)
# Go into your new app folder
cd my-app-name
# Install dependencies
npm install

Connect Dev Store & Run Locally

Terminal — Run Dev Server 
# This command does EVERYTHING:
# - Starts your Next.js dev server
# - Creates a tunnel URL (so Shopify can reach localhost)
# - Installs the app on your dev store
# - Opens the app in your browser
shopify app dev
# On first run, it will ask which store to use.
# Select your development store.
# CLI will print a URL like: https://abc123.trycloudflare.com
✅ What Just Happened?
Shopify CLI automatically created a Cloudflare tunnel from your localhost:3000 to a public HTTPS URL. Shopify can now reach your local app. No Ngrok needed!

Common CLI Errors & Fixes

❌ Error: "shopify: command not found" ▼
Node.js or npm isn't installed, or npm global bin path isn't in your PATH. Fix: Run npm install -g @shopify/cli again. On Mac, try sudo npm install -g @shopify/cli. Restart terminal after.
❌ Tunnel not working / "Could not connect" ▼
Cloudflare tunnel sometimes has issues. Fix: Kill the process (Ctrl+C) and run shopify app dev again. If using Windows, try running terminal as Administrator.
❌ "App not installed on store" error ▼
Your dev store may not have the app installed. Fix: Run shopify app dev — it will ask you to select a store and install the app automatically.
❌ "Missing SHOPIFY_API_KEY" environment variable ▼
Your .env file is missing or incorrect. Run shopify app dev — it auto-generates the .env file. Never commit your .env file to git.
❌ Node version error ▼
Shopify CLI requires Node.js v18+. Fix: Install NVM (Node Version Manager) and run nvm install 20 then nvm use 20.
✓ Pro Tips
  • Always run shopify app dev not npm run dev
  • The CLI handles OAuth tunnel automatically
  • Keep CLI updated: npm update -g @shopify/cli
  • Use pnpm instead of npm — it's faster
✗ Common Mistakes
  • Running npm run dev instead of shopify app dev
  • Committing the .env file to GitHub
  • Using an old Node.js version
  • Not creating a dev store in Partner dashboard first

CHAPTER 07

Hosting & Deployment

Your Shopify app needs to live on a server accessible via the internet — not just your laptop. When a merchant installs your app, Shopify needs to call your URLs. Let's understand hosting options from free to production-grade.

Can You Build for Free?

✅ Yes! Free Options Exist
During development, Shopify CLI gives you a free Cloudflare tunnel. For hosting the final app, Vercel, Railway, and Render all have free tiers. You can build and test a Shopify app with zero hosting cost.

Development: Tunnel URLs

When you run shopify app dev, Shopify CLI creates a tunnel URL — a temporary public HTTPS URL that points to your localhost. This is how Shopify reaches your app while it's running on your computer.

☁️
Cloudflare Tunnel (Built-in)
Auto-created by Shopify CLI. Free. Works instantly. Disappears when you stop the dev server. Best for daily development.
🔗
Ngrok
Alternative tunnel tool. Free tier available. Good if Cloudflare has issues. Run: ngrok http 3000 to get a public URL.

Production Hosting Options

▲
Vercel (Recommended)
Built for Next.js. Free tier. Auto-deploys from GitHub. Zero config. Best choice for most Shopify apps. Add environment variables in dashboard.
🚂
Railway
Simple. Free $5 credit monthly. Supports databases (PostgreSQL included). Great for apps needing a database alongside the server.
🌿
Render
Free tier available (spins down after inactivity). Good for learning. Paid plans for production apps.
🖥️
VPS (DigitalOcean, AWS)
Full control. Starts at $6/month on DigitalOcean. Use for advanced apps with custom setups, background workers, or heavy traffic.

Production Deployment Flow (Vercel)

1. Push code to GitHub
↓
2. Vercel auto-detects push
↓
3. Vercel builds your app
↓
4. Deployed to yourapp.vercel.app
↓
5. Update Shopify Partner App URL → Done ✅

⚠️ Custom Domain for Production
When going to production, buy a domain (e.g., myapp.com) from Namecheap or GoDaddy (~₹800/year) and point it to Vercel. Update the App URL in Shopify Partner dashboard. This looks professional to merchants.
✓ Pro Tips
  • Use Vercel for Next.js — it's made for it
  • Use Railway when you need a database too
  • Always use HTTPS for Shopify apps
  • Set up automatic GitHub deploys from day 1
✗ Common Mistakes
  • Using HTTP instead of HTTPS (Shopify rejects it)
  • Forgetting to update App URL after deploying
  • Not setting environment variables in production
  • Using free Render tier that sleeps — slows merchant installs

Payment & Billing

Intermediate

This is where it gets exciting — making money. Shopify gives you a built-in billing system so you don't need Stripe to charge merchants. Let's understand how to monetize your apps.

Shopify Billing API

The Shopify Billing API lets you charge merchants directly through Shopify. The charge appears on their Shopify invoice — merchants don't need to enter a card. Shopify takes 20% of your revenue and pays you the rest. This is completely automatic.

🔄
Recurring Subscription
Monthly or annual billing. Merchant approves once and is billed automatically. Best for SaaS apps. Example: $9.99/month for your app.
📊
Usage-Based Billing
Charge per action. Example: $0.01 per SMS sent through your app, or $5 per 1000 API calls. Great for apps where usage varies wildly.
💳
One-Time Charge
Charge once for permanent access. Less common for SaaS but useful for setup fees or premium features.

Creating a Subscription Charge

GraphQL — Create App Subscription 
// mutation to create a subscription plan
mutation AppSubscriptionCreate($name: String!, $returnUrl: URL!, $test: Boolean) {
  appSubscriptionCreate(
    name: $name
    returnUrl: $returnUrl
    test: $test
    lineItems: [{
      plan: {
        appRecurringPricingDetails: {
          price: { amount: 9.99, currencyCode: USD }
          interval: EVERY_30_DAYS
        }
      }
    }]
  ) {
    appSubscription {
      id
      status
    }
    confirmationUrl  ← redirect merchant here to approve billing
    userErrors {
      field
      message
    }
  }
}

While Shopify Billing is easiest for charging merchants inside Shopify, you can also use Stripe or Razorpay for:

  • Charging customers (end-users) inside your app's own dashboard
  • Standalone SaaS products that aren't exclusively for Shopify merchants
  • Indian merchants who prefer UPI or net banking

Build Your First Shopify App

Beginner — Hands-On

Theory is nothing without practice. Let's build a real, working Shopify app from scratch. This tutorial creates a simple app that displays your store's products in a beautiful Polaris table inside the Shopify Admin.

What You'll Build
A Shopify embedded app that reads all products from a store and shows them in a table inside the Admin dashboard. Simple but real — it uses OAuth, GraphQL, Polaris, and Next.js.

Note: Shopify CLI uses Remix by default
The current Shopify CLI template uses Remix (not Next.js App Router). Remix is similar to Next.js. All concepts are the same — loaders fetch data server-side, actions handle form submissions. The architecture is identical.

Step 4: Fetch Products from Shopify API

app/routes/app._index.jsx 
import { useLoaderData } from "@remix-run/react";
import { json } from "@remix-run/node";
import { authenticate } from "../shopify.server";
import { Page, Layout, Card, DataTable } from "@shopify/polaris";
// ─── LOADER: runs on server, fetches products ─────────────
export const loader = async ({ request }) => {
  const { admin } = await authenticate.admin(request);
  const response = await admin.graphql(
    `#graphql
      query {
        products(first: 20) {
          edges {
            node {
              id
              title
              totalInventory
              status
            }
          }
        }
      }
    `
  );
  const { data } = await response.json();
  return json({ products: data.products.edges });
};
// ─── COMPONENT: displays products in Polaris table ────────
export default function Index() {
  const { products } = useLoaderData();
  const rows = products.map(({ node }) => [
    node.title,
    node.status,
    node.totalInventory
  ]);
  return (
    <Page title="My Products">
      <Layout>
        <Layout.Section>
          <Card>
            <DataTable
              columnContentTypes={['text', 'text', 'numeric']}
              headings={['Product', 'Status', 'Inventory']}
              rows={rows}
            />
          </Card>
        </Layout.Section>
      </Layout>
    </Page>
  );
}

Advanced Shopify Apps

Advanced

Once you can build a basic app, the next step is going deeper. Advanced Shopify apps handle real business complexity — syncing data, integrating with external systems, and extending Shopify's core functionality.

Webhooks (Advanced Usage)

At the beginner level, webhooks "notify you when something happens". At the advanced level, you use them for real-time data pipelines.

Webhook Handler — Order Created 
// app/routes/webhooks.jsx
import { authenticate } from "../shopify.server";
import db from "../db.server";
export const action = async ({ request }) => {
  // IMPORTANT: Always verify the webhook signature first!
  const { topic, shop, payload } = await authenticate.webhook(request);
  switch (topic) {
    case "ORDERS_CREATE":
      // New order! Save to DB, send notification, sync to ERP
      await db.order.create({
        data: {
          shopifyId: payload.id,
          shop: shop,
          total: payload.total_price,
          email: payload.email
        }
      });
      break;
    case "PRODUCTS_UPDATE":
      // Product changed — update your cache or external system
      await syncProductToERP(payload);
      break;
  }
  return new Response("OK");
};

Types of Advanced Apps

🔄
Inventory Sync Apps
Sync Shopify stock with warehouses (3PLs like ShipBob), Amazon, or ERP systems. Uses webhooks + scheduled jobs + Inventory API.
🏭
ERP Integrations
Connect Shopify to SAP, Tally, Zoho. Sync orders, invoices, customer data bidirectionally. High-value custom projects.
🤖
AI-Powered Apps
AI product descriptions, chatbots, recommendation engines using OpenAI/Gemini API. Hot category in 2025.
📊
Analytics Apps
Custom dashboards, cohort analysis, LTV calculations, inventory forecasting. Merchants pay well for data insights.
🛒
Checkout Extensions
Add custom UI blocks to Shopify checkout — upsells, gift messages, address validators. Requires Shopify Plus or checkout extensibility.
⚙️
Shopify Functions
Serverless logic that runs INSIDE Shopify's infrastructure. Custom discounts, shipping rules, payment filters. Written in Rust or JavaScript.

Background Jobs & Queues

Some tasks can't run during an HTTP request — they take too long. Background jobs handle heavy processing asynchronously.

Example: A merchant clicks "Sync 50,000 products to ERP". You can't make 50,000 API calls in one request. Instead, you create a background job that processes in the background while the merchant keeps using the app.

⏱️
BullMQ (Node.js)
Job queue library for Node.js using Redis. Schedule jobs, retry failed jobs, process in parallel. Industry standard.
🔧
Trigger.dev
Modern background jobs platform. Free tier. Works great with Next.js and Shopify apps. No Redis needed.
✓ Pro Tips
  • Always verify webhook HMAC signatures
  • Use Shopify Functions for checkout logic
  • AI apps are the hottest category now
  • Checkout extensions = Shopify Plus premium clients
✗ Common Mistakes
  • Not handling webhook duplicate delivery
  • Doing heavy work synchronously (causes timeouts)
  • Not using rate limiting when calling Shopify API
  • Ignoring Shopify API rate limits (leaky bucket)

SaaS (Software as a Service) is the ultimate goal — you build an app once, merchants pay you every month, you earn while you sleep. This is how real wealth is built in the Shopify ecosystem.

What is Micro-SaaS?

Micro-SaaS is a small, focused software product that solves one specific problem really well, built and maintained by 1–3 people. It's the perfect model for individual Shopify developers.

📢

Sends order updates via WhatsApp. Merchants love this. ₹499–₹2,999/month. Potential: 200+ merchants = ₹5–60 lakh/month.

📝

Generates product descriptions using AI. $29/month. OpenAI API cost is tiny. Pure profit after first few merchants.

App Store Strategy

  1. Find a gap — look at 1-star reviews of existing apps. What are merchants complaining about? Build that.
  2. Build MVP in 2–4 weeks — simple version, 3 core features
  3. Submit for App Store review — takes 1–5 days. Follow Shopify's requirements carefully.
  4. Get first 10 reviews — ask early users for honest reviews. These make or break discoverability.
  5. Optimize listing — great screenshots, clear value proposition, demo video
  6. Iterate based on feedback — add features merchants request. Each update = better retention.
